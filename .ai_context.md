```text

protected-src/      TypeScript source for the gated direction prototypes + content (NOT bundled into dist/)

tests/              automated validation — never delete to force a passing build
rules/              repository constraints — treat as binding, not suggestions
scripts/            automation and tooling (scripts/ci/ enforces repo structure)
docs/agents/        focused agent instruction sub-files
```

The hosting `public` directory in `firebase.json` is `dist/` — the Vite
build output. `vite.config.ts` defines the multi-page entries (gate `/`,
`/menu`, `/d/{1,2,3}`) and externalizes only an explicit allowlist of
hosts: `fonts.googleapis.com` (Google Fonts CSS stylesheets) and
`fonts.gstatic.com` (the underlying woff2 font binaries the stylesheets
reference). All JavaScript is bundled — Firebase comes from npm
into the auth chunk, React + ReactDOM come from npm into each protected
module, and there are no `<script src="https://...">` tags loading
runtime code. The migration is being staged through #20 — phases 1
(toolchain), 2 (TS port + npm Firebase), 3 (protected sources to TS +
esbuild build), and 4 (loader swapped to blob-URL dynamic import +
unpkg React/Babel UMD scripts retired) are merged. Phases 5–6 ship
the responsive refinements from #11.

## Deploy Tooling

This repo follows the AI Agent Tooling Standard's keyless deployment model.
Human maintainers authenticate through a shared 1Password-backed GCP ADC
source credential, then deploy through short-lived service account
impersonation. CI prefers Workload Identity Federation.

### Scripts

| Script | Canonical source | Installed to | Purpose |
|---|---|---|---|
| `gcloud` | `scripts/gcloud/gcloud` | `~/.local/bin/` | Local wrapper that refreshes access tokens from a 1Password-backed source credential |
| `op-firebase-deploy` | `scripts/firebase/op-firebase-deploy` | `~/.local/bin/` | Deploy by impersonating the deployer service account |
| `op-firebase-setup` | `scripts/firebase/op-firebase-setup` | `~/.local/bin/` | One-time setup: creates deployer SA, grants IAM roles, configures impersonation |

### Cloudflare cache purge (auto on deploy)

The custom domain `tadlockpsychiatry.com` is fronted by Cloudflare.
Every Hosting deploy auto-purges the Cloudflare edge cache via a
`postdeploy` hook in `firebase.json` that runs `scripts/cf-purge.sh`.

- Token: `op://Private/4x6wslp3f6pal5t6h3jhhe63ie/credential` —
  the **All Domains—Cache Purge API token** (scoped to Cache Purge
  only, works against any zone in the account).
- Zone: `b4a043e316d4c2b76af6fff655c0f4a6` for tadlockpsychiatry.com
  (hard-coded as default in `scripts/cf-purge.sh`; override with
  `ZONE_ID=...` env).
- Script fails SOFT — a Cloudflare API hiccup logs a warning but
  doesn't fail the deploy. Set `CF_PURGE_FAIL_HARD=1` to flip.
- `DRY_RUN=1 bash scripts/cf-purge.sh` previews without calling the
  API. `SKIP=1 bash scripts/cf-purge.sh` no-ops (useful in CI).
